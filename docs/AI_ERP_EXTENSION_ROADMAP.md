# AI / ERP Extension Roadmap

Date: 2026-05-04

## Purpose

This document describes the Odoo/ERP workflow direction for the Monte Carlo real-estate analytics dashboard. It is a roadmap and contract-framing note, not a claim that production integration exists today.

The current package includes a local deterministic business-summary JSON export generated from the existing model, a local Odoo/ERP-style handoff payload, and dry-run mapping notes. These artifacts show how the dashboard could feed reporting, API, AI/MCP, or ERP/Odoo-style handoff workflows without changing the financial model or claiming production business-system integration.

## Current Capability

The repository currently provides:

- a Streamlit dashboard for local scenario review
- the `monte_carlo_model.py` Monte Carlo engine
- core return and risk metrics such as IRR, NPV, cash-on-cash, equity multiple, DSCR, debt yield, LTV, and exit cap
- Trace / Explain support and export surfaces
- tests and audit artifacts for bounded validation evidence
- a deterministic local export payload at `artifacts/integration_demo/sample_business_summary.json`
- a local Odoo/ERP-style dry-run handoff payload with no network calls

The local payload is generated by `python scripts/export_demo_business_summary.py`. It uses the existing model only and does not contact external services. Sandbox validation evidence may be documented separately; no live or production Odoo/ERP integration is included in the default local demo path.

## Future Architecture

```text
dashboard/model core
  -> structured export contract
  -> optional API layer
  -> optional AI/MCP tools
  -> optional ERP/Odoo handoff
```

The engineering principle is to keep the model as the source of truth and expose only carefully bounded outputs. Future systems should consume structured summaries, report bundles, or explicitly versioned schemas rather than scraping UI state.

## Why Odoo/ERP Would Matter Later

The current repository does not implement production Odoo, ERP, CRM, SAP, MCP, or hosted API integration. Those remain roadmap and future-direction items, not default application behavior.

A future business-system workflow could pull approved property, lease, tenant, invoice, operating-cost, debt, or project records into the model instead of relying on manual re-entry. After a simulation is reviewed, a future system could push outputs back as reports, notes, attachments, approval tasks, risk summaries, or audit-trail records.

The current structured export payload is the bridge point for that future workflow. It demonstrates how model outputs can be shaped for downstream reporting or business process handoff without claiming production ERP sync.

A local Odoo/ERP-style handoff demo payload and dry-run mapper extend this bridge by reshaping the business-summary export into future record, attachment, review-task, and audit-trail concepts. Sandbox validation evidence may be documented separately, but this package should not be described as live or production Odoo/ERP integration.

## Candidate Future Tool Surface

If a tool-calling or MCP layer is ever built, candidate tool names could include:

- `run_simulation`
- `get_metrics_summary`
- `explain_p50_irr`
- `compare_scenarios`
- `generate_risk_summary`
- `export_report_bundle`

These names are future interface candidates only. They are not implemented as live tools in this package.

## ERP / Odoo Handoff Direction

A later ERP/Odoo path should start as a reporting or document handoff workflow:

- export a reviewed metrics payload
- attach a scenario summary to an opportunity, account, project, or document record
- route a report bundle into a business review process
- keep the Monte Carlo model as the specialized analytics layer

This package includes a local Odoo/ERP-style dry-run handoff path, but it does not include default business-system sync, production workflow automation, or production external authentication.

## Implemented Now

Implemented in this package:

- local deterministic business-summary export payload
- fixed-seed base scenario generation
- JSON contract with package metadata, scenario metadata, core metrics, risk flags, validation boundary, future targets, and explicit non-claims
- a pytest contract check for payload shape and forbidden positive claims
- local Odoo/ERP-style dry-run handoff payload
- dry-run workflow mapping notes for future record, attachment, review-task, and audit-trail concepts

## Not Implemented

Not included in this package:

- production Odoo connector deployment
- default ERP, CRM, or SAP sync
- MCP server
- OpenAI or AI-agent integration
- hosted API
- cloud deployment
- autonomous decision workflow
- production Odoo integration

## Validation Gates Before Any Live Claim

Before making any live integration claim, the project would need:

- stable versioned request and response schemas
- schema tests for every exported field
- deterministic fixture coverage for sample payloads
- authentication and authorization design
- secret handling and environment separation
- audit logging for workflow actions
- integration tests against a non-production target system
- security review for external data movement
- clear user-facing limits around financial interpretation and human review

Until those gates are complete, this repository should be described as a local portfolio demo with an integration-ready export artifact and dry-run workflow mapping, not as an integrated production enterprise system.
